# Contributing to Maestro

Thanks for your interest in contributing. This document covers the basic development flow during **v0.1**.

## Setup

```bash
pnpm install           # installs dependencies across all workspaces
pnpm build             # builds all packages via tsup
pnpm test              # Vitest in each package
pnpm typecheck         # tsc --noEmit in each package
pnpm lint              # ESLint + strict rules
pnpm format:check      # checks if code is in line with Prettier
```

Requirements:

- Node.js >= 20 (LTS 24 recommended; default target for builds)
- pnpm >= 9 (the repo pins via `packageManager` in `package.json`)

## PR Flow

1. Open a branch from main following the pattern `lucassampaiodegois1/<dsft-id>-<slug>` (the name generated by Linear works).
2. Make small, descriptive commits. We use Conventional Commits (https://www.conventionalcommits.org/) (`feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`, `ci:` etc.).
3. Ensure that `pnpm lint`, `pnpm typecheck`, `pnpm test` and `pnpm build` pass before opening the PR. The CI will run them automatically.
4. **Add a changeset** if the change affects any publishable package (see below).
5. Open the PR pointing to main. Link to the Linear issue (e.g., Refs DSFT-###).

## Changesets

We use [Changesets](https://github.com/changesets/changesets) for versioning and automated changelogs.

### When to add a changeset

Whenever the change affects observable behavior or the public API of any `@maestro/*` package. For internal changes (test refactors, docs that don\'t affect users, CI tweaks), it is not required.

If in doubt, add one.

### How to add

```bash
pnpm changeset
```

The interactive CLI will:

1) Ask which packages were affected — mark with space.
2) Ask the bump type for each: patch (fix), minor (feature-compatible), major (breaking change).
3) Ask for a short description in natural language.

It generates a file in `.changeset/<random-id>.md`. Commit this file along with the PR.

### Linked packages

During v0.1, all @maestro/* packages are linked: any bump propagates the same version to all. This simplifies the first release and avoids misaligned versions while the product is still unstable. In v0.2+ we can separate release cycles by package.

### Release

- `pnpm version-packages` consumes pending changesets, bumps versions and generates CHANGELOG.md per package.
- `pnpm release` runs `pnpm build` and publishes to npm with `changeset publish`.

The process typically runs via [changesets/action] in CI (the release workflow will be added in DSFT-98).

## Code style

- Strict TypeScript. Avoid `any`, `as unknown as ...`, and non-null assertions (`!`) in production code.
- ESM-only. Imports should use `.js` extensions in relative paths (the tsconfig uses `moduleResolution: bundler`, but we prefer explicit for compatibility).
- No redundant comments. Comment reasons and trade-offs, not what the code is doing.
- No `console.log` in final code. Use the logger when available (post DSFT-78).

For details on why, read Maestro — Fundamentals and Architecture document at Linear.

## Reporting problems

During v0.1 the official issue tracker is Linear (Dreamsoft team, Maestro project). Open issues with enough context to reproduce: command run, `maestro --version`, Node version, OS, relevant logs.

On GitHub, templates in `.github/ISSUE_TEMPLATE/` serve as entry points for reports from outside the team. PRs use the template in `.github/PULL_REQUEST_TEMPLATE.md`.

## Code of conduct

This project adheres to the Contributor Covenant 2.1. In short:

- Treat everyone with respect. No discrimination, harassment or hostility.
- Technical discussions are about code and decisions—not people.
- Report violations to `lucas.gois@dreamsoft.dev` (private). We will not retaliate against those reporting in good faith.

The full Covenant applies even if not duplicated here. In case of conflict, the official version at https://www.contributor-covenant.org/version/2/1/code_of_conduct/ is the authoritative.
